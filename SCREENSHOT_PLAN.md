# Screenshot Plan

Last updated: 28 March 2026

This file defines the planned screenshot set for App Store and Google Play submission. It is a production prep document, not proof that the final screenshots have already been captured.

## 1. Goal

The screenshot set should communicate the core v1 value of Tassulokero in the clearest possible order:

1. all important pet information in one place
2. reminders and routines stay visible
3. shared care works for family and caretakers
4. the app supports updates, media, and shared care
5. breeder-related tracking is available when needed

## 2. Primary Screenshot Sequence

Recommended order for the first 5 screenshots:

### Screenshot 1

Headline:

- All your pet's important info in one place

Screen:

- Home screen with one clear active pet card and upcoming highlights

Purpose:

- establish the app as a real pet dashboard, not just a note app

### Screenshot 2

Headline:

- Health, care notes and routines kept together

Screen:

- Pet detail with overview, health and care sections visible

Purpose:

- show the structured record-keeping value clearly

### Screenshot 3

Headline:

- Never miss the next important task

Screen:

- Reminders screen with today, upcoming and overdue groups

Purpose:

- show the reminder value quickly

### Screenshot 4

Headline:

- Share the right info with the right people

Screen:

- Sharing screen showing family, family admin and caretaker roles

Purpose:

- show the strongest differentiator in the product

### Screenshot 5

Headline:

- Keep a shared care history with updates and media

Screen:

- Pet updates or media view with shared activity

Purpose:

- show that the app supports real day-to-day collaboration

## 3. Optional Extended Screenshot Set

Use these if store slots allow or if the launch emphasizes broader scope:

### Screenshot 6

Headline:

- Breeder records and heat cycle tracking in the same app

Screen:

- Breeder access or heat cycle section

## 4. Platform Notes

Apple App Store:

- keep copy shorter and cleaner
- favor strong feature hierarchy over too many labels
- avoid overcrowded screens

Google Play:

- can tolerate slightly denser explanatory overlays
- still keep the first 3 screenshots very direct

## 5. Device Size Plan

Prepare at minimum:

- iPhone 6.7" screenshots
- iPhone 6.5" fallback if needed
- Android phone screenshots

Optional later:

- tablet screenshots if tablet support is actively marketed

## 6. UI State Requirements Before Capture

Before taking final store screenshots, confirm:

- final app icon is in place
- final splash asset is in place
- no placeholder media labels are visible
- no debug or mock helper text is visible
- no unfinished feature disclaimers are visible in screenshot builds
- sample pet names, notes and handles are polished
- colors and typography match final launch styling

## 7. Content Setup for Screenshot Session

Create one clean demo dataset with:

- one main pet with complete profile data
- one secondary pet for multi-pet proof
- realistic reminders in different states
- one family member and one caretaker
- a few polished updates with media
- one breeder-linked pet if breeder screenshot is included

## 8. Visual Style Notes for Overlays

Recommended overlay style:

- short headline only
- no paragraph text
- high contrast background or gradient strip
- consistent placement across the whole set
- one message per screenshot

Avoid:

- long explanations
- too many badges
- tiny text
- screens that require zooming to understand

## 9. Capture Checklist

Before capture:

- prepare clean demo data
- choose final screenshot order
- confirm which screenshots are used for Apple and which for Google Play
- hide dev tools and unfinished UI
- verify navigation state and spacing on device

After capture:

- crop and export to store-required sizes
- add text overlays if needed
- review each image for placeholder content
- verify privacy and branding consistency

## 10. What Still Needs to Happen

This plan does not mean screenshots are already complete.

Still required before the checklist item can be honestly marked as fully done in a final release sense:

- run the app in final screenshot state
- capture real device or simulator screenshots
- export store-ready assets
- review them against the final listing copy and launch branding
